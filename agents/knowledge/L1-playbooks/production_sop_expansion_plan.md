# 生产级 SOP 扩展路线图

目标：
- 让 OpenClaw 不只是“能跑命令”
- 而是逐步拥有一批真正可投入生产的 SOP
- 这些 SOP 要能被 `main` 编排，必要时分配给 `life-assistant`、`executor`、`designer`、`engineer`

## 选择原则

优先做这类 SOP：
- 交付物明确
- 触发频率高
- 用户一眼能判断是否成功
- 可沉淀成模板与知识
- 可以部分自动化，然后逐步升级到高自动化

避免优先做这类 SOP：
- 输入长期不稳定
- 输出无法验证
- 完全依赖临场创意、没有交付格式
- 需要大量手工介入却没有反馈闭环

## P0：应先落地

1. `XHS 选题雷达`
   - 用于每天/每周从小红书或外部信号生成选题池
   - 交付物：选题榜单、优先级、建议角度

2. `XHS 内容多平台改写包`
   - 把一份源材料改写成多平台发布包
   - 交付物：标题、正文、短视频脚本、摘要版

3. `资料交接包`
   - 把项目目录整理成交接文档
   - 交付物：README、目录说明、风险点、下一步

4. `研究资料矩阵`
   - 把检索结果整理成研究问题-来源-证据矩阵
   - 交付物：可直接阅读和继续补充的资料矩阵

## P1：适合继续增强

1. `服务异常排查与恢复建议`
2. `技能缺口检查与安装计划`
3. `周报/周复盘自动组合产物`
4. `内容素材沉淀与再利用索引`

## 每条 SOP 的最低标准

- 必须有明确输入
- 必须有明确输出位置
- 必须有可验证结果
- 必须能写案例或教训到知识库
- 必须能被路由到合适 agent

## 自动化成熟度分级

### L1
- 可手动触发
- 有固定输出格式

### L2
- 可自动归档产物
- 可自动写案例

### L3
- 可根据反馈自动重跑或生成进化任务

### L4
- 可定时/事件驱动运行
- 可形成稳定流水线

## 当前推荐推进顺序

1. 先把 P0 的 4 条全部做成 SOP 文件
2. 再给其中 2 条补显式执行器
3. 再补任务验证、反馈、自进化
4. 再做事件驱动 / 定时化
